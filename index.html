RippleNet: Propagating User Preferences on the Knowledge Graph for Recommender Systems

Posted on 2020-09-05 In KnowledgeGraph

文章主要解决什么问题

利用 KnowledgeGraph 解决推荐系统中的点击率预测。使用 KnowledgeGraph 的 unified 方法（即混合了 embedding-based 和 path-based）。

解决的思路是什么

在知识图谱上，基于用户点击历史获得用户的偏好。考虑到相近的 item 可能有关联，所以用类似于水波涟漪的方式，从几个点击过的 item 向外传播，一层一层的获取用户不同方面的 latent vector。
也就是说，对于每个用户，RippleNet 都将用户的历史兴趣作为 KnowledgeGraph 中的种子集，然后沿着 KnowledgeGraph 的链接迭代地向外传播与拓展，以发现用户对某个候选 item 的潜在兴趣。
再基于 latent vector 与 item 的 embedding-vector 预测点击率。

核心的知识点有哪些

模型分块说明

Ripple Set

根据用户点击历史可以定义出 relevant entity 和 ripple set。

  • 用户对 entity 的潜在兴趣由用户的历史偏好激活并沿着 KG 逐层传播
  • 随着跳数的增加，用户的潜在偏好强度减弱

Preference Propagation

首先获取用户历史点击过的 entity 的 vector；将需要预测的 item v 进行 embedding 得到 vector。
然后通过 pi=softmax⁡(vTRihi)=exp⁡(vTRihi)∑(h,r,t)∈Su1exp⁡(vTRh)p_{i}=\operatorname{softmax}\left(\mathbf{v}^{\mathrm{T}} \mathbf{R}_{i} \mathbf{h}_{i}\right)=\frac{\exp \left(\mathbf{v}^{\mathrm{T}} \mathbf{R}_{i} \mathbf{h}_{i}\right)}{\sum_{(h, r, t) \in \mathcal{S}_{u}^{1}} \exp \left(\mathbf{v}^{\mathrm{T}} \mathbf{R} \mathbf{h}\right)}pi​=softmax(vTRi​hi​)=∑(h,r,t)∈Su1​​exp(vTRh)exp(vTRi​hi​)​ 计算 item v 在 relation R 的情况下和历史 entity i 的相似度。
这样加权之后，就可以获得这一层的输出 ou1=∑(hi,ri,ti)∈Su1piti\mathbf{o}_{u}^{1}=\sum_{\left(h_{i}, r_{i}, t_{i}\right) \in \mathcal{S}_{u}^{1}} p_{i} \mathbf{t}_{i}ou1​=∑(hi​,ri​,ti​)∈Su1​​pi​ti​ 。这意味着用户 u 对 item v 在本层获取到的偏好。可以通过迭代的方式多扩展几层，每层都会获得这样一个偏好向量。最后将各层的偏好向量加和，得到用户 u 对 item v 的偏好 embedding.
将上述偏好 embedding 和 item embedding vector 内积并使用 softmax 得到点击概率。

其他

本文亮点：

  • 不需要像一些 path-based 方法那样手工构造 meta-path / meta-graph。
  • 将 KGE 与 KG 的图结构结合起来进行兴趣传播。
  • 因为路径的缘故，可解释性较好。

Meta-Graph Based Recommendation Fusion over Heterogeneous Information Networks

Posted on 2020-09-04 In KnowledgeGraph

文章主要解决什么问题

利用知识图谱，采用 path-based 的方法实现推荐系统的评分预测。

解决的思路是什么

所有的信息与交互历史保存在异构图（or 知识图谱）中，根据不同的 meta-graph 子图可以提取出不同的特征。

  1. 基于 meta-graph 的相似度
    在知识图谱中人工定义出一些规范的 meta-graph。每个类型的 entity 之间都有一个转移矩阵，那么 user类 和 item类 之间的这一个子图就可以表述为一连串矩阵乘积。若是有多条路径，可以采取 element-wise product 融合。有了这样的关系，就可以计算出每个 user 和每个 item 的相似度。所有的 user 和所有的 item 的相似度构成了在第 l 个 meta-graph 下的相似度矩阵。当有 L 个 meta-graph，就有 L 个这样的相似度矩阵。

  2. 基于 meta-graph 的 Latent Features
    基于矩阵分解的思想，根据 object function 可以将每个矩阵分解为 user 矩阵与 item 矩阵，得到每个 user 和 item 的向量。

    min⁡U,B12∥PΩ(UB⊤−R)∥F2+λu2∥U∥F2+λb2∥B∥F2\min _{\mathbf{U}, \mathbf{B}} \frac{1}{2}\left\|P_{\Omega}\left(\mathbf{U B}^{\top}-\mathbf{R}\right)\right\|_{F}^{2}+\frac{\lambda_{u}}{2}\|\mathbf{U}\|_{F}^{2}+\frac{\lambda_{b}}{2}\|\mathbf{B}\|_{F}^{2}U,Bmin​21​∥∥∥​PΩ​(UB⊤−R)∥∥∥​F2​+2λu​​∥U∥F2​+2λb​​∥B∥F2​

  3. 基于 L 个 meta-graph 的矩阵分解结果实现评分预测
    对于每一个观察到的 user item 对，可以用一个向量表示。

    xn=ui(1),…,ui(l),…,ui(L)⏟L×Fbj(1),…,bj(l),…,bj(L)⏟L×F\mathbf{x}^{n}=\underbrace{\mathbf{u}_{i}^{(1)}, \ldots, \mathbf{u}_{i}^{(l)}, \ldots, \mathbf{u}_{i}^{(L)}}_{L \times F} \underbrace{\mathbf{b}_{j}^{(1)}, \ldots, \mathbf{b}_{j}^{(l)}, \ldots, \mathbf{b}_{j}^{(L)}}_{L \times F}xn=L×Fui(1)​,…,ui(l)​,…,ui(L)​​​L×Fbj(1)​,…,bj(l)​,…,bj(L)​​​

    L 表示有 L 个 meta-graph, F 表示矩阵分解的维数。
    采用一种 FM 的方法，进行一阶和二阶的特征交叉，预测最终结果

    y^n(w,V)=w0+∑i=1dwixin+∑i=1d∑j=i+1d⟨vi,vj⟩xinxjn\hat{y}^{n}(\mathbf{w}, \mathbf{V})=w_{0}+\sum_{i=1}^{d} w_{i} x_{i}^{n}+\sum_{i=1}^{d} \sum_{j=i+1}^{d}\left\langle\mathbf{v}_{i}, \mathbf{v}_{j}\right\rangle x_{i}^{n} x_{j}^{n}y^​n(w,V)=w0​+i=1∑d​wi​xin​+i=1∑d​j=i+1∑d​⟨vi​,vj​⟩xin​xjn​

    target function 为

    min⁡w,V∑n=1N(yn−y^n(w,V))2\min _{\mathbf{w}, \mathbf{V}} \sum_{n=1}^{N}\left(y^{n}-\hat{y}^{n}(\mathbf{w}, \mathbf{V})\right)^{2}w,Vmin​n=1∑N​(yn−y^​n(w,V))2

  4. 正则化
    采用 group lasso regularization 的正则化方式。

核心的知识点有哪些

  • meta-graph 提取信息
  • MF 矩阵分解
  • FM 特征交叉

其他

跟 meta-path 相比，meta-graph 是有优势的。meta-path必须是序列的一条路径，而实际上 entity 的关系可能是网状的。针对这种情况，其他方法可能将这样的网状关系拆分成一条条独立的 meta-path 来处理，用一个加权参数求和，就不能很好的捕获不同路径之间的关系。

首先暴搜肯定会超时，需要考虑剪枝策略。
剪枝策略是这样的：从栈中最后一个元素的位置到当前元素位置之间，如果已经出现过当前元素了，则不需要进入搜索，因为已经在出现过的位置搜索过了。举个例子
2 4 5 7 8 9 7 10
当前栈中元素是 [2 3 5] 搜索到第二个 7 的时候是需要剪枝剪掉的，因为它所包含的情况已经在第一个 7 出现的时候搜索过了。
有了剪枝策略就很好写代码了。

leetcode-1568-使陆地分离的最少天数

Posted on 2020-09-03 In leetcode
  1. 判断是否只有一个岛，如果是则需要进一步处理，否则就意味着不需要操作，直接返回 0.
  2. 如果只有一个岛，则有两种情况。首先，最坏的情况下，也最多只需要两天就可以完成了（随意切下来一个角就分离了一块）。否则，就意味着存在着一块地连接着两边。可以先将相邻的陆地之间连边，这样问题就转化为了求割点。使用 tarjan 判断割点是否存在即可。

下面简要回顾一下 tarjan
当当前节点是起始节点时，如果它的子树超过两颗，那么它是割点。当当前节点 u 不是起始节点，那么如果它的某个子节点 v 满足 low[v] >= dfn[u] ，说明可以往上翻到 u 的父节点，那么 u 就不是割点，否则 u 是割点。
这题主要是要自己建立边，有点麻烦。
